⏺ Here's a summary of the project:

  ---
  Minum is a minimalist Java web framework (v9.0.0, com.renomad:minum on Maven
  Central) for building server-side-rendered web applications. Its defining
  philosophy: zero external dependencies, plain Java only — no reflection, no
  annotations, no magic.

  Goals

  - Long-term maintainability and quality over fast startup / feature richness
  - 100% branch+statement test coverage, ~100% mutation test strength
  - Small footprint: ~6,500 lines of production code vs. Javalin's 141K or
  Spring Boot's 1M+

  What it provides (all built-in)

  ┌──────────────┬───────────────────────────────────────────────────────────┐
  │    Module    │                       What it does                        │
  ├──────────────┼───────────────────────────────────────────────────────────┤
  │ web/         │ HTTP/HTTPS server, routing (WebFramework),                │
  │              │ request/response, body parsing                            │
  ├──────────────┼───────────────────────────────────────────────────────────┤
  │ database/    │ In-memory + disk-persistent DB (DbEngine2 recommended; Db │
  │              │  simpler)                                                 │
  ├──────────────┼───────────────────────────────────────────────────────────┤
  │ logging/     │ Lazy-closure-based logger with levels: debug, trace,      │
  │              │ audit, asyncError                                         │
  ├──────────────┼───────────────────────────────────────────────────────────┤
  │ templating/  │ {{ key }} string templates, with XSS-safe helpers         │
  ├──────────────┼───────────────────────────────────────────────────────────┤
  │ htmlparsing/ │ Lightweight HTML parser for use in tests and server logic │
  ├──────────────┼───────────────────────────────────────────────────────────┤
  │ security/    │ IP banning (TheBrig)                                      │
  ├──────────────┼───────────────────────────────────────────────────────────┤
  │ queue/       │ Background ActionQueue for parallel processing            │
  ├──────────────┼───────────────────────────────────────────────────────────┤
  │ testing/     │ TestFramework with assertEquals, assertTrue, TestLogger   │
  ├──────────────┼───────────────────────────────────────────────────────────┤
  │ utils/       │ StringUtils, LRUCache, CryptoUtils, FileUtils, etc.       │
  └──────────────┴───────────────────────────────────────────────────────────┘

     Minum Codebase Review: Verified Issues & Fixes

     Context

     Three exploration agents reviewed all production source files in the Minum
     web framework. I verified each high-severity finding against the actual
     source code, filtering out false positives. This plan covers only
     confirmed, real issues worth fixing.

     ---
     Tier 1 — HIGH (bugs that can crash or corrupt in production)

     1. Unhandled NumberFormatException in Headers.contentLength()

     File: src/main/java/com/renomad/minum/web/Headers.java:118
     Impact: A malformed Content-Length header (e.g. "abc") crashes request
     handling with an unhandled exception.
     Fix: Wrap Integer.parseInt(cl.getFirst()) in try-catch, default to -1.

     2. Race condition: dataSet.add() outside synchronized block

     File: src/main/java/com/renomad/minum/database/AbstractDb.java:252-260
     Impact: In addToIndexes(), computeIfAbsent is synchronized but
     dataSet.add(dbData) at line 260 is not, while removeFromIndexes() IS fully
     synchronized. Concurrent add/remove can corrupt index sets.
     Fix: Move dataSet.add(dbData) inside the existing synchronized (this)
     block.

     3. Non-volatile stop flag — visibility race

     File: src/main/java/com/renomad/minum/queue/ActionQueue.java:40
     File: src/main/java/com/renomad/minum/logging/LoggingActionQueue.java:26
     Impact: boolean stop is read/written from different threads without
     synchronization. The JMM does not guarantee visibility — a thread could
     enqueue work after stop, or never see the stop.
     Fix: Change to volatile boolean stop in both classes.

     ---
     Tier 2 — MEDIUM (resource leaks, correctness, thread-safety)

     4. Resource leak in Response.compressBody()

     File: src/main/java/com/renomad/minum/web/Response.java:432-443
     Impact: GZIPOutputStream is not in try-with-resources. If gos.write(body)
     throws, the stream leaks.
     Fix: Wrap in try-with-resources.

     5. LRUCache not thread-safe, used from concurrent FileReader

     File: src/main/java/com/renomad/minum/utils/LRUCache.java
     File: src/main/java/com/renomad/minum/utils/FileReader.java:33-34, 77
     Impact: LRUCache extends LinkedHashMap (not thread-safe). FileReader does
     containsKey then get (TOCTOU race) and put from multiple request threads.
     Fix: Wrap in Collections.synchronizedMap() in getLruCache(), or use
     ConcurrentHashMap with a different eviction strategy.

     6. RingBuffer.containsAt() — NoSuchElementException on miss

     File: src/main/java/com/renomad/minum/utils/RingBuffer.java:107-124
     Impact: Uses while(true) calling iterator.next() without checking
     hasNext(). If the searched pattern extends beyond the buffer,
     NoSuchElementException is thrown.
     Fix: Replace while(true) with while(iterator.hasNext()), return false after
      loop.

     7. Query string early-return discards valid parameters

     File: src/main/java/com/renomad/minum/web/RequestLine.java:209
     Impact: if (equalSignLocation <= 0) return Map.of() — encountering one
     malformed pair (e.g. foo=bar&bad&baz=qux) drops all params including valid
     ones already parsed.
     Fix: continue instead of return Map.of() (skip the bad pair, keep going).

     ---
     Tier 3 — LOW (minor correctness, defense-in-depth)

     8. Boundary index off-by-one

     File: src/main/java/com/renomad/minum/web/BodyProcessor.java:130
     Impact: indexOfBoundaryKey > 0 should be >= 0. Extremely unlikely in
     practice since content-type headers always have multipart/form-data; before
      boundary=.
     Fix: Change to >= 0.

     9. Boundary value not trimmed at semicolons

     File: src/main/java/com/renomad/minum/web/BodyProcessor.java:132
     Impact: If content-type has trailing params after boundary (e.g.
     boundary=foo; charset=utf-8), the boundary includes ; charset=utf-8.
     Fix: Trim at first ; or whitespace after extracting.

 Architecture investigation
 --------------------------

 Here are the architectural friction points I found in Minum, presented as module-deepening candidates:

   ---
   1. WebFramework.java — The God Class (952 lines)
   - Cluster: WebFramework, Server, Response, TheBrig, static file serving, BodyProcessor
   - Why coupled: One class owns HTTP protocol handling (keep-alive, compression, content-length), request routing, static file serving, pre/post middleware, and security
   integration
   - Category: Shared-state coupling — multiple concerns co-own the request lifecycle
   - Test impact: Static methods extracted purely for testability (potentiallyCompress, determineIfKeepAlive, confirmBodyHasContentType) — real bugs hide in the sequencing between
    them inside httpProcessing()

   2. Database Dual-Implementation + Late Index Registration
   - Cluster: Db, DbEngine2, AbstractDb, DatabaseConsolidator, index registration pattern
   - Why coupled: Two implementations with no polymorphic selection; indexes registered post-instantiation (db.registerIndex(...) called separately after construction);
   DbFileConverter for migration
   - Category: Temporal coupling — object usable before it's fully configured
   - Test impact: Both implementations tested separately; two-phase initialization tested implicitly; no test can verify "forgot to register index" failure mode at compile time

   3. Request Body Lifecycle — Split Ownership
   - Cluster: Request, BodyProcessor, WebFramework.httpProcessing()
   - Why coupled: Three classes co-own body state: Request has the hasStartedReadingBody flag, BodyProcessor does actual parsing, WebFramework sequences the calls — all
   runtime-checked, none compile-time enforced
   - Category: Implicit protocol ordering — the rules exist in code comments and runtime exceptions
   - Test impact: BodyProcessorTests, RequestTests, and WebFrameworkTests test these in isolation; integration scenario (interrupted multipart upload, access-after-read) requires
   full stack

   4. Response Factory Method Proliferation (8+ variants)
   - Cluster: Response.java, callers across WebFramework, user endpoint code
   - Why coupled: 8+ static factory methods (buildResponse, buildStreamingResponse, buildLeanResponse, range variants) with no shared builder; callers must know which to use
   - Category: Interface shallowness — the method count IS the complexity
   - Test impact: Each variant tested independently; combination/upgrade paths (e.g., "stream this unless it's small, compress if large") only tested via full WebFramework tests

   5. HTML Parser State Machine
   - Cluster: HtmlParser.java, State inner class, TagInfo, ParseNodeV
   - Why coupled: Character-by-character state machine with a 12-char ring buffer, flags (isInsideTag, isInsideComment, isInsideScript), and line tracking all tangled in one
   mutable State object
   - Category: Dense implementation hiding a shallow interface — hard to extend or modify
   - Test impact: High unit test coverage, but state transitions are tested via strings, not by driving the state machine directly

     ---
     High-Level Analysis of Minum
   
     The project is impressively disciplined — 100% test coverage, zero
     dependencies, clear documentation, and a small surface area. That said, a few
     patterns stand out when measured against modern Java and web framework
     conventions.
   
     1. Null Returns Instead of Optional
   
     The codebase has 16 return null sites across production code and zero uses of
     Optional. The most impactful example is findExactlyOne() in
     AbstractDb.java:414, which returns null when nothing is found. This is a
     well-known footgun in Java APIs — callers get no compile-time reminder to
     handle the absent case.
   
     This is a conscious philosophical choice, but it works against the framework's
      own "no surprises" ethos. Every null return is a latent NullPointerException
     for a user who forgets to check. Optional was designed precisely for this —
     signaling "might not be there" in the type signature.
   
     2. Concurrency Strategy is Mixed
   
     AbstractDb uses ConcurrentHashMap for the main data map but plain HashMap for
     registeredIndexes and partitioningMap (lines 81, 87, 95-96). The index
     operations then use synchronized(this) blocks (lines 252, 274) — but only
     around part of the operation. For example in addToIndexes (line 252-254), the
     computeIfAbsent is synchronized but the dataSet.add(dbData) on line 260 is
     not, creating a window where two threads could race on the same HashSet.
   
     This mixing of three concurrency strategies (ConcurrentHashMap, synchronized
     blocks, and unsynchronized HashMap) in one class is hard to reason about and
     audit.
   
     3. DbData Serialization is Fragile
   
     The DbData contract asks users to manually implement serialize()/deserialize()
      with pipe-delimited, URL-encoded strings. This is error-prone:
     - Field ordering must be kept in sync between the two methods
     - Adding/removing a field is a breaking migration with no versioning
     - No schema evolution story
   
     For a framework that values longevity, this is a real maintenance risk for
     users. Even a simple record-based approach with named fields would be more
     resilient.
   
     4. Context is a Service Locator
   
     Context holds logger, executor, constants, FullSystem, and ActionQueueState —
     and is passed into nearly every constructor. This is essentially a service
     locator pattern, which has known downsides:
     - Hides actual dependencies (a class takes Context but only needs ILogger)
     - Makes unit testing harder (you must construct a full Context even if you
     only need one piece)
     - The setFullSystem() mutator (line 58) creates a temporal coupling — Context
     is partially constructed until FullSystem calls back
   
     Given the framework's philosophy of explicitness, direct constructor injection
      of only the needed dependencies would be more aligned.
   
     5. Constants Exposes All Fields as public final
   
     Constants has ~20 public fields with no encapsulation (lines 54-212). While
     final prevents reassignment, this means:
     - Every consumer is coupled to the field name and type
     - You can't add validation, transformation, or lazy computation later without
     a breaking change
     - keystorePassword is a public field on a freely-passed object — this is a
     security smell
   
     This class would be a natural record (it's already effectively immutable), or
     at minimum should use accessor methods.
   
     6. No Middleware Composition Model
   
     The pre-handler/last-minute-handler system (registerPreHandler,
     registerLastMinuteHandler) is limited to one of each. This means:
     - You can't compose concerns (auth + logging + CORS + rate-limiting as
     separate middleware)
     - Users must manually combine logic into a single monolithic pre-handler
     - This is a significant gap compared to the middleware chain pattern used by
     virtually every modern web framework (Express, Koa, Javalin, etc.)
   
     A List<PreHandler> executed in order would preserve simplicity while enabling
     composition.
   
     7. processDataIndex Does a Linear Scan
   
     In AbstractDb.java:190:
     boolean dataEntryExists = data.values().stream()
         .anyMatch(x -> x.getIndex() == newData.getIndex());
   
     This is an O(n) scan on every update operation, even though data is a
     Map<Long, T> — a simple data.containsKey(newData.getIndex()) would be O(1).
     For a framework that advertises 2M ops/sec, this is a surprising miss.
   
     8. Abstract Class Where Interface Would Suffice
   
     DbData<T> is an abstract class with four abstract methods and no concrete
     behavior or state. This forces single inheritance on every data type users
     create. An interface (potentially sealed) would be less constraining and more
     idiomatic for modern Java, especially since Java 21 is the target.
   
     9. Modern Java Features Underused
   
     The project targets Java 21 but doesn't leverage several features that would
     improve the code:
     - Sealed interfaces — IResponse has exactly two expected shapes (body vs
     streaming); sealing would make this explicit and enable pattern matching
     - Pattern matching in switch/instanceof — would clean up several type-check
     chains
     - Records — used in some places but TemplateSection, Inmate, and many data
     carriers are still classes with manual equals/hashCode
   
     10. No Structured Error Responses
   
     When an endpoint throws, WebFramework returns a plain-text "Server error: 
     <correlationId>". There's no way for users to customize error rendering (HTML
     error page, JSON error body, etc.) without wrapping every handler in
     try-catch. A global error handler hook (similar to the pre-handler pattern)
     would be a natural addition.
